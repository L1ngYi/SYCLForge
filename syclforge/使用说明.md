# SYCLForge 中文使用说明

这份说明面向在远端 A100 机器上测试 `syclforge` 的场景。当前工具的目标是：读取
`cpu_sycl/gemm_*.cpp` 中固定形状的 SYCL GEMM，按 CudaForge 的模式做“编译验证、
计时、NCU 指标反馈、LLM 修复/优化、保存 best kernel”的自动迭代。

## 1. 拉取代码

```bash
git clone https://github.com/L1ngYi/SYCLForge.git
cd SYCLForge
```

## 2. 环境要求

至少需要：

- Python 3.10+，建议使用 CudaForge 原来的 conda 环境。
- `numpy`。
- 支持 NVIDIA CUDA backend 的 SYCL 编译器，例如 Intel oneAPI/LLVM SYCL 的 `icpx`
  或支持 `-fsycl-targets=nvptx64-nvidia-cuda` 的 `clang++`。
- NVIDIA A100 驱动/CUDA runtime。
- `ncu` 可选，但推荐安装。没有 `ncu` 时可以用 `--no-ncu` 跑 latency-only 流程。

如果使用原 CudaForge 环境，可以先试：

```bash
conda env create -f environment.yml
conda activate CudaForge
pip install numpy openai pandas matplotlib
```

如果远端已经有可用环境，只需要保证：

```bash
python -c "import numpy; print(numpy.__version__)"
icpx --version
ncu --version
```

## 3. A100 SYCL CUDA backend 设置

如果远端机器和下面类似，`icpx` 默认不在 `PATH`，但 `/home/l1ngyi/env_sycl.sh`
会把自编译的 DPC++/LLVM CUDA backend 加进来：

```bash
source /home/l1ngyi/env_sycl.sh
```

加载后先确认：

```bash
which clang++
which icpx || true
sycl-ls
ncu --version
```

如果你不想每次手动 `source`，也可以让 SYCLForge 的子进程自动 source 这个脚本：

```bash
export SYCL_ENV_SCRIPT=/home/l1ngyi/env_sycl.sh
```

SYCLForge 也会自动尝试查找当前目录、仓库目录、用户 home 目录，以及当前目录/仓库目录的父目录中的
`env_sycl.sh`。因此在 `/home/l1ngyi/workspace/SYCLForge` 下运行时，通常能自动找到
`/home/l1ngyi/env_sycl.sh`；但显式 `source` 或设置 `SYCL_ENV_SCRIPT` 最稳。

另外建议显式设置：

```bash
export ONEAPI_DEVICE_SELECTOR=cuda:gpu
export SYCL_TARGETS=nvptx64-nvidia-cuda
export SYCL_CUDA_ARCH=sm_80
```

如果 `icpx` 不在 `PATH`，但 `clang++` 在 `/home/l1ngyi/dpcpp_cuda/llvm_build/bin`
中，也可以直接指定：

```bash
export SYCL_COMPILER=/home/l1ngyi/dpcpp_cuda/llvm_build/bin/clang++
```

如果你的 SYCL 安装需要额外编译参数，可以追加：

```bash
export SYCL_EXTRA_FLAGS="..."
```

## 4. 先跑一个不调用 LLM/NCU 的烟测

建议先跑最小测试，确认入口、case 发现、harness 生成、编译路径是否正常：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --case-stem gemm_128_128_128 \
  --round 1 \
  --no-llm \
  --no-ncu
```

如果成功，会在 `syclforge_runs/.../summary.csv` 里看到结果。如果编译失败，也会正常
写出失败日志，不会直接崩掉。重点看：

```text
syclforge_runs/<本次运行>/<case>/evaluation/round_000_seed.json
syclforge_runs/<本次运行>/<case>/harness/round_000.cpp
```

## 5. 跑带 NCU 的 seed 评测

确认编译没问题后，再打开 NCU：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --case-stem gemm_128_128_128 \
  --round 1 \
  --no-llm
```

这一步会用 24 个 CudaForge 同款 Nsight Compute 指标生成 profile feedback。如果远端没有
权限读取 GPU performance counters，`ncu` 可能报权限错误；可以临时继续用 `--no-ncu`，
或者按服务器权限策略配置 NCU。

正常调优建议让 GPU compute mode 保持 `Default`。如果机器被配置成 `Exclusive_Process`，
并且自动 NCU 报 `CUDA_ERROR_DEVICE_UNAVAILABLE`，可以临时加 `--isolated-benchmark`
让 benchmark 阶段在独立子进程里执行，避免主 Python 进程持有 CUDA context。

## 6. 跑单个 case 的自动调优

配置好 LLM API 后，可以跑 6 轮示例：

```bash
export OPENAI_API_KEY=你的_key

python -m syclforge.main_sycl cpu_sycl \
  --case-stem gemm_1024_1024_1024 \
  --gpu A100 \
  --server_type openai \
  --model_name o3 \
  --round 6
```

如果不用 OpenAI，而是本地/vLLM/sglang，可以沿用 CudaForge 的参数，例如：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --case-stem gemm_1024_1024_1024 \
  --gpu A100 \
  --server_type vllm \
  --server_address 127.0.0.1 \
  --server_port 8000 \
  --model_name deepseek-ai/deepseek-coder-6.7b-instruct \
  --round 6
```

### Tensor Core / joint_matrix 实验模式

默认 prompt 走稳妥的 SIMT SYCL GEMM 优化。如果想让 LLM 主动尝试 A100 Tensor Core，
可以加 `--tensor-core`。默认 `--tensor-core-mode auto` 会先按 shape 路由：

```text
strong/try: M、N 足够大，K 不太小，并且 M%16==0、N%16==0、K%8==0
avoid: K 太小、M/N 太小、或 tile 对齐不适合 Tensor Core
```

被路由到 `strong/try` 的 case 会走双轨候选：SIMT lane 和 Tensor Core lane 都会生成并评测，
最终仍由 benchmark/NCU 结果选 best。若要强制尝试 Tensor Core，可以加：

SIMT lane 会始终从当前最好的 SIMT/seed 代码出发，Tensor lane 会从当前最好的 Tensor/overall
代码出发，避免两条 lane 互相污染。

```bash
--tensor-core-mode force
```

若有一份已验证的 joint_matrix baseline，也可以作为 skeleton mutation lane：

```bash
--tensor-core-skeleton /path/to/best.cpp
```

示例：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --case-stem gemm_2048_2048_2048 \
  --round 4 \
  --server_type deepseek \
  --model_name deepseek-v4-pro \
  --max_tokens 16000 \
  --tensor-core \
  --rtol 2e-2 \
  --atol 5e-2
```

TF32 Tensor Core 不是严格 FP32 乘法。如果候选编译和运行都成功，但只因为 `rtol=1e-3`
和 `atol=1e-3` 没过正确性，可以按实验目标放宽阈值，例如：

```bash
--rtol 2e-2 --atol 5e-2
```

SYCLForge 会把当前 `--rtol/--atol` 写进 Tensor Core prompt，避免 LLM 继续按默认
`1e-3` 约束生成代码。

开启后，SYCLForge 会先编译并运行一个 TF32 `joint_matrix` canary。只有 canary 编译和运行
都通过，Judge/Optimizer prompt 才会允许并鼓励使用：

```text
<sycl/ext/oneapi/matrix/matrix.hpp>
sycl::ext::oneapi::experimental::matrix::joint_matrix
joint_matrix_load / joint_matrix_mad / joint_matrix_store
```

Tensor Core 模式会自动使用隔离 benchmark 子进程。原因是 experimental `joint_matrix`
候选如果触发 SYCL runtime/CUDA backend 原生断言，可能直接 abort 当前进程；隔离后会记录为
该轮失败，而不会中断整次调优。计分仍然使用 kernel event profiling 时间，不包括子进程启动时间。

如果 canary 失败，默认会退回普通 SIMT prompt，并把探针日志写到：

```text
<run>/tensor_core_probe.json
<run>/tensor_core_probe/*_canary.cpp
```

如果你希望 canary 失败时直接停止，可以额外加：

```bash
--require-tensor-core
```

验证是否真的用到了 Tensor Core，看每轮 evaluation JSON 里的 NCU 指标：

```text
sm__inst_executed_pipe_tensor.avg.pct_of_peak_sustained_active
```

如果这个值仍接近 0，说明生成代码虽然可能编译通过，但没有真正打到 tensor pipe。

## 7. 批量跑 20 个 GEMM

先建议少量测试：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --first-n 3 \
  --gpu A100 \
  --server_type openai \
  --model_name o3 \
  --round 4
```

确认稳定后跑全部：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --gpu A100 \
  --server_type openai \
  --model_name o3 \
  --round 6
```

## 8. 保存 best kernel 到单独目录

默认不会覆盖 `cpu_sycl` 原文件。若希望把每个 case 的 best kernel 输出到一个目录：

```bash
python -m syclforge.main_sycl cpu_sycl \
  --first-n 20 \
  --gpu A100 \
  --server_type openai \
  --model_name o3 \
  --round 6 \
  --write-back-dir syclforge_best
```

输出会是：

```text
syclforge_best/gemm_128_128_128.cpp
syclforge_best/gemm_256_256_256.cpp
...
```

## 9. 输出文件怎么看

每次运行会生成一个目录：

```text
syclforge_runs/<timestamp>_<case或batch>_<model>/
```

重要文件：

```text
summary.csv                         # 所有 case 的汇总
summary.json                        # 所有 case 的详细汇总
<case>/summary.json                 # 单个 case 的详细结果
<case>/best.cpp                     # 当前 case 的 best kernel
<case>/code/round_XXX_*.cpp         # 每轮候选源码
<case>/evaluation/round_XXX_*.json  # 每轮编译、正确性、性能、profile 结果
<case>/harness/round_XXX.cpp        # 自动生成的 benchmark harness
<case>/profile/round_XXX_ncu.csv    # NCU 原始 CSV，若 profiling 成功
<case>/llm_io/*.txt                 # 每轮 prompt 和模型回复
```

## 10. 常见问题

### 编译器不认识 `-fsycl-targets=nvptx64-nvidia-cuda`

说明当前 `clang++/icpx` 不是支持 CUDA backend 的 SYCL 编译器。请加载正确的 oneAPI/LLVM
SYCL 环境，或设置：

```bash
export SYCL_COMPILER=/path/to/支持CUDA后端的/icpx
```

### `ncu` 报权限错误

先用 `--no-ncu` 跑通编译和正确性；之后再配置 NCU performance counter 权限。

### `ncu` 下出现 `CUDA_ERROR_DEVICE_UNAVAILABLE`

如果 `--no-ncu` 能跑通，但打开 NCU 后看到类似：

```text
CUDA_ERROR_DEVICE_UNAVAILABLE
UR_RESULT_ERROR_UNKNOWN
```

说明 SYCL CUDA backend 在 Nsight Compute 启动的 profile 进程里没有成功创建 CUDA
device/context。新版脚本会自动依次尝试：

```text
--replay-mode=application
--replay-mode=kernel
```

Nsight Compute 2023.1.1 不支持 `--replay-mode=disabled`，所以脚本不会使用它。

如果仍失败，可以先手动确认：

```bash
RUN=syclforge_runs/<本次运行目录>
EXE=$RUN/<case>/profile/round_000_profile

$EXE
ncu --replay-mode=application --target-processes=all --set full "$EXE"
nvidia-smi
nvidia-smi -q -d COMPUTE
```

重点检查是否有其他进程占用 GPU、GPU 是否处于 exclusive process 模式、MIG/MPS 配置是否
限制了当前进程。若必须在 `Exclusive_Process` 下运行，可以给 SYCLForge 加：

```bash
--isolated-benchmark
```

### LLM 输出不是合法 C++/SYCL

程序会把 parse error 记录在：

```text
<case>/llm_io/round_XXX_*_parse_error.json
```

并沿用上一轮候选，避免整个 batch 中断。

### 想只测试编译和性能，不调用模型

使用：

```bash
--no-llm
```

### 想只测试编译和 event timing，不跑 NCU

使用：

```bash
--no-ncu
```
